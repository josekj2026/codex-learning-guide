What Codex is for

Codex is OpenAI's coding agent for software development. Use it to write, understand, review, debug, test, and automate work across your projects.

Local work

Work in your project

Local threads read and edit your files, run commands, and report what changed. Use them when you want tight feedback in the repo on your machine.

Isolated work

Use worktrees for experiments

Worktree threads isolate changes from your current checkout, which is useful for parallel tasks, risky refactors, or trying an idea without disturbing your main work.

Remote work

Delegate to cloud tasks

Cloud threads run in an isolated environment after your repo is connected. They are useful for parallel work, PR follow-ups, or tasks started away from your machine.

The step-by-step path

Move through these in order. Each step gives you a working habit, not just a feature name.

Choose the right surface

Start where the work naturally lives. Use the app for parallel project work, the IDE extension when open files matter, and the CLI when you want terminal-first control.

  • Use local mode for direct edits in the current project.
  • Use worktree mode for isolated experiments.
  • Use cloud mode when the work can run remotely from a connected repository.

Prompt with four ingredients

Give Codex the goal, context, constraints, and definition of done. This keeps it scoped and makes the result easier to verify. 

Goal: Fix the settings save bug.
Context: The issue appears on /settings. Start with @src/settings.
Constraints: Keep the API shape unchanged.
Done when: Reproduce the bug, patch it, add a regression test, and run the smallest relevant checks.

Use planning when the task is fuzzy

For complex or unclear work, ask Codex to plan first. In the Codex app, toggle plan mode with /plan or Shift+Tab. Ask it to interview you when the idea is still rough.

  • Ask for assumptions and risks before implementation.
  • Approve the plan only after the scope feels reviewable.
  • Turn long work into a clear /goal when there is a measurable finish line.

Let Codex investigate before editing

Good Codex work usually starts with reading files, searching the codebase, running a repro, or inspecting terminal output. Ask for that explicitly when correctness matters. 

Before editing, find the relevant call sites and explain the likely cause.
Then make the smallest patch and tell me exactly how you verified it.

Verify every meaningful change

Ask Codex to run tests, lint, type checks, or a manual reproduction. For web apps, use the in-app browser to preview local routes, leave visual comments, and verify rendered behavior.

  • Prefer the smallest relevant test suite first.
  • Use the integrated terminal for dev servers and checks.
  • Ask Codex to report commands, results, and any checks it could not run.

Review the diff like a teammate

Use the review pane to inspect uncommitted changes, last-turn changes, or branch changes. Leave inline comments on specific lines, then ask Codex to address them with a narrow scope. 

Address the inline comments only.
Keep the public API unchanged and rerun the same checks afterward.

Save repeated guidance in AGENTS.md

When you find yourself repeating setup commands, test expectations, repo conventions, or review rules, put them in AGENTS.md. Keep it short and practical.

  • Add repo layout and important folders.
  • List build, test, lint, and format commands.
  • Record the review standards and "done" criteria your team expects.

Grow into skills and MCP

Use skills for repeatable workflows with rich instructions. Use MCP when Codex needs live tools or external systems such as documentation, issue trackers, design files, logs, or GitHub.

  • Start with AGENTS.md for repo rules.
  • Create a skill when the workflow repeats across tasks.
  • Add MCP when the workflow needs outside data or actions.

Prompt patterns

These are templates you can adapt. The strongest prompts give Codex a way to check its own work.

Explain unfamiliar code

Read @path/to/file and nearby callers.
Explain the request flow, the data that is validated, and the risky assumptions.
End with a short list of files I should inspect next.

Fix a reproducible bug

Bug: [what happens]
Repro: [numbered steps]
Constraints: [what must not change]
Done when: reproduce it, fix it, add a regression test if feasible, and run checks.

Add a feature safely

Implement [feature] in [area].
Follow existing patterns in [files].
Keep the change small. Add tests for the new behavior.
Before finishing, review the diff for regressions.

Use the browser for UI work

Use @Browser to open http://localhost:3000/[route].
Verify the [state] on desktop and mobile.
Fix only the visible issue and keep the existing layout structure.

A 7-day practice plan

Spend 20 to 40 minutes per day. The point is to build judgment about when to steer, when to let Codex work, and how to verify the result.

Daily missions

  • Day 1: Ask Codex to explain a small project and list the main files.
  • Day 2: Give it a tiny bug or TODO and require a verification command.
  • Day 3: Use /plan on a fuzzy task before allowing edits.
  • Day 4: Ask Codex to add or improve one test around existing behavior.
  • Day 5: Use the review pane and leave one inline comment for Codex to fix.
  • Day 6: Write a starter AGENTS.md with commands and conventions.
  • Day 7: Repeat a previous task with the new guidance and compare the result.

What to notice

Codex improves when the environment is explicit. Track what it needed to ask, what it guessed correctly, what checks caught issues, and which instructions you repeated. 

After each session, ask:
1. What did I forget to tell you?
2. Which command verified the result?
3. What should go into AGENTS.md for next time?

Customization ladder

Start simple. Add durable machinery only when it removes repeated friction.

First

Prompt

Use the prompt for one-off task details, temporary constraints, exact files, acceptance criteria, and the checks you want run this time.

Next

AGENTS.md

Use AGENTS.md for repo conventions, commands, review expectations, architectural boundaries, and definitions of done.

Then

config.toml

Use configuration for durable defaults such as model choice, reasoning, sandbox settings, approvals, profiles, feature flags, and MCP setup.

Repeatable

Skills

Use skills when a workflow needs reusable instructions, examples, scripts, references, or a clear trigger phrase.

Connected

MCP

Use MCP when Codex needs live capabilities from external systems: docs, tickets, designs, logs, browsers, or hosted development tools.

Ongoing

Automations

Use automations for recurring checks, reminders, monitors, or thread wake-ups where Codex should come back on a schedule.

Before you trust the result

A good Codex session ends with evidence. This checklist keeps the finish line concrete.

Accept the change when

  • The requested behavior is implemented or the limitation is clearly explained.
  • The diff is scoped to the task and follows local patterns.
  • Relevant tests, lint, type checks, or manual repro steps were run.
  • Codex reported any checks it could not run.
  • You reviewed the diff and asked for follow-up fixes where needed.

Ask for another pass when

  • The answer is broad but lacks file references or verification.
  • The patch changes unrelated files or invents a new pattern.
  • The test plan is vague, skipped, or disconnected from the bug.
  • Codex seems blocked by missing context you can provide.
  • You repeated an instruction that belongs in AGENTS.md.
